# Extraction plan for python/bindings/system/base/system.cpp

PLAN ONLY. This document changes no code. It identifies four extraction candidates
in `python/bindings/system/base/system.cpp` (1805 lines), states the current code ranges,
the dependencies and the risks, then proposes a sequence of bit-identical PRs.

---

## 0. Context

`python/bindings/system/base/system.cpp` is the only translation unit of the `System` runtime facade. It
contains, all mixed together:
- loading a `.so` via `dlopen` (three paths: JIT, AOT, native);
- the elliptic solver and its wiring (lazy construction, eps/kappa fields);
- the block registry (`Impl::Species`, `sp`) and its type-erased closures;
- the time step (`step`, `step_cfl`, `step_adaptive`).

The size makes navigation costly and reviews long. The goal is to split it without
any observable behavior change.

---

## 1. NativeLoader

### What it owns

Everything that touches loading a `.so` generated by the DSL:

- `add_dynamic_block`: dlopen, reading `pops_model_nvars`, dispatch to `push_dynamic<NV>`,
  template `Impl::push_dynamic<NV>` (host closures `host_residual`, `max_speed`, `advance`,
  `add_poisson`);
- `add_compiled_block`: dlopen, reading the AOT symbols (`pops_compiled_residual/advance/
  max_speed/poisson_rhs/naux`), building the marshaling closures, `read_block_meta`;
- `add_native_block`: dlopen with RTLD_GLOBAL promotion, ABI safeguard (`pops_native_abi_key`
  vs `abi_key()`), call to `pops_install_native`;
- utilities shared by all three: `host_residual<NV>`, `limited_slope`, `read_block_meta`,
  `parse_var_set`, `split`, `BlockMeta`.

### Current ranges (approximate)

| Subsection | Lines |
|---|---|
| `limited_slope` + `host_residual<NV>` (host template) | 111-197 |
| `role_index`, `split`, `BlockMeta`, `parse_var_set`, `read_block_meta` | 198-266 |
| `Impl::push_dynamic<NV>` (JIT block) | 733-858 |
| `System::add_dynamic_block` | 985-1015 |
| `System::add_compiled_block` | 1017-1153 |
| `System::add_native_block` | 1155-1241 |

Estimated total: ~350 lines.

### Dependencies

- `dlopen`/`dlsym`/`dlclose` (POSIX) + `dlfcn.h`;
- `IModel<NV>` (`runtime/dynamic_model.hpp`) for the JIT path;
- `GridContext` (`runtime/block_builder.hpp`) for the AOT/native closures;
- `System::install_block` (must stay in system.cpp or live in a shareable header);
- `System::ensure_aux_width` (modifies the shared aux channel, see SystemBlockStore);
- `abi_key()` (line 98, exported symbol).

### Risks

- LOW. None of these paths is on the GPU/MPI hot path. The closures capture `P`
  (`Impl*` pointer) and the type-erased lambdas of `push_dynamic` contain `copy_state`/
  `write_state` which stay in `Impl` (where they must be exposed).
- Point to watch: `push_dynamic<NV>` calls `P->ensure_aux_width` and `P->write_state` /
  `P->copy_state`. If NativeLoader is extracted out of `Impl`, these accesses must go through
  an interface or stay friends of `Impl`.
- `add_native_block` issues a `dlopen(RTLD_GLOBAL)` that modifies the visibility of the
  current module. This behavior must stay atomic and must not be reordered relative to the
  ABI safeguard.

## 2. SystemFieldSolver

### What it owns

Everything that builds and uses the elliptic solver:

- lazy configuration (`ensure_elliptic`, `Impl::p_rhs`, `p_solver`, `p_bc`, `p_wall`,
  `p_wall_radius`, `p_eps_`, `has_eps_field_`, `p_eps_field_`, `has_eps_xy_field_`,
  `p_eps_x_field_`, `p_eps_y_field_`, `has_kappa_field_`, `p_kappa_field_`,
  `std::optional<std::variant<GeometricMG, PoissonFFTSolver>> ell_`);
- installing the operator fields: `apply_epsilon_field`, `apply_epsilon_anisotropic_field`,
  `apply_reaction_field`;
- internal accessors: `ell_rhs()`, `ell_phi()`, `ell_solve()`;
- computing the electric field and populating the aux: `solve_fields()` (the body);
- public APIs: `System::set_poisson`, `System::set_epsilon_field`,
  `System::set_epsilon_anisotropic_field`, `System::set_reaction_field`,
  `System::solve_fields()`, `System::potential()`.

| `System::potential()` | 1793-1803 |

Estimated total: ~120 lines + Impl members.

### Dependencies

- `GeometricMG` and `PoissonFFTSolver` (included headers);
- `apply_bz()` and `apply_te()`: called from `ensure_aux_width` and `solve_fields` ->
  dependency with SystemBlockStore. `solve_fields` calls `apply_te()` after
  `ell_solve()`: the order is semantically important (T_e computed AFTER the new phi,
  B_z preserved);
- `comm::n_ranks()` (FFT MPI safeguard);
- `wall_active()` -> `detail::wall_predicate` (`runtime/wall_predicate.hpp`);
- `device_fence()` (after `ell_solve`, before the grad phi derivation): a hot spot for
  device bugs. DO NOT reorder this fence.
- `fill_boundary` and `fill_ghosts` (aux halo filling at the end of `solve_fields`).

### Risks

- MEDIUM. `solve_fields` is called every step from `step`/`step_cfl`/`step_adaptive`.
  It is a hot path in the frequency sense (but not in the GPU kernel sense: the `for_each_cell`
  loop lives in `GeometricMG`).
- The `device_fence()` between `ell_solve()` and the grad phi derivation is a correctness
  invariant on GPU (without this fence, the V-cycle kernel is not finished when `phi_mf`
  is read). Any extraction must preserve this order.
- The guard `if (rhs.local_size() == 0) return` in the MPI loops is critical (avoids
  `fab(0)` on a rank without a box). To be audited if the body is moved.

### Dependencies

- `dispatch_model` (`runtime/model_factory.hpp`) in `add_block`;
- `make_block`, `make_max_speed`, `make_poisson_rhs`, `make_cell_convert`, `block_n_ghost`
  (`runtime/block_builder.hpp`) in `add_block`;
- `CondensedSchurSourceStepper` (`coupling/condensed_schur_source_stepper.hpp`) in
  `set_source_stage` and `run_source_stage`;
- `CoupledSourceKernel` (`coupling/coupled_source_program.hpp`) in `add_coupled_source`;
- `for_each_cell` in the coupling lambdas;
- `sum` (`mesh/mf_arith.hpp`) in `mass`.
- `ensure_aux_width` + `apply_bz`/`apply_te`: potential circular dependencies with
  SystemFieldSolver (both read/write the `aux` channel). To be handled by an interface or by
  co-locating in the same file.

### Risks

- MEDIUM-HIGH. `add_ionization`, `add_collision`, `add_thermal_exchange` build
  lambdas capturing `P->sp[ia].U.fab(0).array()`: the fab is referenced DIRECTLY. If
  `sp` is reallocated between registration and application (push_back), the lambdas
  would become invalid. This is an EXISTING risk (not introduced by the extraction), but
  the extraction must not hide it.
- `add_coupled_source` and `add_ionization` use `fab(0)` without a `local_size()` guard for
  the coupling lambdas (unlike `apply_bz`/`solve_fields` which iterate over
  `local_size()`). This is not a bug here (System has a single box, round-robin, so the
  owner always has `fab(0)`) but it is fragile if the layout changes.
- The coupled sources are applied AFTER `step`/`step_cfl` via `apply_couplings`. The order
  "transport then couplings" is a semantic convention to preserve.

## 4. SystemStepper

### What it owns

The time step logic:

- `System::step(dt)`;
- `System::advance(dt, nsteps)`;
- `System::step_cfl(cfl)`;
- `System::step_adaptive(cfl)`.

### Current ranges (approximate)

| Subsection | Lines |
|---|---|
| `System::step` | 1642-1659 |
| `System::advance` | 1660-1662 |
| `System::step_cfl` | 1663-1701 |
| `System::step_adaptive` | 1702-1738 |

Total: ~100 lines.

### Dependencies

- `solve_fields()` (Impl call): called at the head of each step;
- `Impl::stride_due`: filter by block;
- `s.advance(s.U, eff_dt, s.substeps)`: type-erased closure captured in Species;
- `run_source_stage(s, eff_dt)`: OPT-IN Schur stage (no-op if nullptr);
- `apply_couplings(dt)`: post-transport coupled sources;
- `s.max_speed(s.U)` in `step_cfl`/`step_adaptive` to compute the per-block CFL dt.

### Risks

- LOW. These four functions are readable orchestrators with no direct access to the fabs.
  They call the blocks of SystemFieldSolver (via `P->solve_fields()`) and SystemBlockStore
  (via `P->sp`). The extraction is clean.
- Point to watch: the CFL formula in `step_cfl` (substeps-aware, post-#121) is documented
  and commented; do not simplify it by mistake during the move.
- `step_adaptive` computes the substeps on the fly (`ceil(stride * w_b / w_min)`): the multirate
  semantics are subtle (see comment line 1726). Verify that the comments follow.

## 5. Recommended PR sequence

Principle: each PR keeps the CI tests green (bit-identical) and changes no behavior.
Sequence from least dependent to most dependent.

### PR-1: extract NativeLoader

WRITE-SET:
- New file `python/system_native_loader.cpp` (or `python/native_loader.cpp`);
- `python/bindings/system/base/system.cpp`: remove the moved functions, add an include if needed;
- `python/CMakeLists.txt`: add the new file to the `_pops` target.

Moved content: `host_residual`, `limited_slope`, `split`, `BlockMeta`, `parse_var_set`,
`read_block_meta`, `Impl::push_dynamic<NV>`, `add_dynamic_block`, `add_compiled_block`,
`add_native_block`.

DEPENDENCIES TO EXPOSE: `Impl::copy_state`, `Impl::write_state`, `Impl::ensure_aux_width`
must remain accessible from the new file (via friend, via an Impl header, or by
moving them into a shareable header). The least invasive solution: place `push_dynamic`
in a header `python/impl_dynamic.hpp` included by both TUs, or implement it in the
new file passing `Impl*` explicitly (already the case: `push_dynamic(Impl* P, ...)`).

ACCEPTANCE TESTS:
- `test_dynamic_model` (type-erased model == static Euler);
- `test_compiled_model_parity` (AOT native == native block);
- `test_dsl_*` (Python suite);
- all current CI tests (Release + MPI + Kokkos).

RISK: LOW. No hot path, no direct access to the GPU fabs. The only point: keep the
symbols `POPS_EXPORT` (`install_block`, `grid_context`, `ensure_aux_width`) in `system.cpp`
so the native loaders resolve them.

### PR-2: extract SystemFieldSolver

WRITE-SET:
- New file `python/system_field_solver.cpp`;
- `python/bindings/system/base/system.cpp`: remove the moved Impl methods and members.

Moved content: elliptic Impl members (`p_rhs`, ..., `ell_`), `ensure_elliptic`,
`apply_epsilon_field`, `apply_epsilon_anisotropic_field`, `apply_reaction_field`, `ell_rhs()`,
`ell_phi()`, `ell_solve()`, `solve_fields()` (Impl body), `System::set_poisson`,
`System::set_epsilon_field`, `System::set_epsilon_anisotropic_field`,
`System::set_reaction_field`, `System::solve_fields()`, `System::potential()`,
`System::ensure_aux_width` (delegation).

CRITICAL DEPENDENCIES:
- `apply_bz()` and `apply_te()` stay in SystemBlockStore but are called from
  `ensure_aux_width` and `solve_fields`. Solution: expose them as public methods of
  `Impl` (they already are) and call them via `P->apply_bz()`.
- `device_fence()` between `ell_solve()` and the grad phi derivation: this order must stay
  atomic. Do not refactor it into two separate calls.
- The `local_size() == 0` guard in the derivation and B_z/T_e population loops:
  verify it is preserved.

ACCEPTANCE TESTS:
- `test_bindings.py::test_poisson_*` (variable eps, screened, anisotropic);
- `test_mpi_system_solve_fields_np{1,2,4}` (bit-identical np=1/2/4);
- `test_system_abstraction`, `test_system_coupler`.

RISK: MEDIUM. `solve_fields` is on the hot path (called per step) but the body contains
no critical loop: it is an orchestrator. The real risk is the accidental
reordering of the `device_fence`.

DEPENDENCIES: all dependencies are methods of `Impl` (`solve_fields`, `sp`,

## 7. Precondition common to all the PRs

Before PR-1, create a header `python/impl_fwd.hpp` (or equivalent) that exposes:
- The complete definition of `Impl` (or the members needed by the external TU);
- The `Impl` methods called cross-TU.

This avoids placing method definitions in a header and generating ODR violations.
A more conservative alternative: use the friend + forward declaration technique in `system.hpp`,
and leave `Impl` defined only in `system.cpp`. The methods of the auxiliary TUs then receive
`Impl*` and have access via `friend` declared in `System::Impl`.

Bit-identical verification (harness): each PR compares `max|U_before - U_after|` on a reference
case (diocotron n=64, 20 steps, np=1) and verifies `dmax == 0.0` before merging.
